# エンジニアポートフォリオサイトの技術構成と設計に関する包括的調査レポート

## 採用担当者が5分で判断する技術選択の境界線

エンジニアのポートフォリオサイトは、技術力のアピールと実用性のバランスが重要な課題となっています。本調査では、2024-2025年の最新トレンドと採用担当者の視点を交えながら、適切な技術構成について包括的に分析しました。最も重要な発見は、**採用担当者の86%がポートフォリオサイトを確認するものの、平均滞在時間はわずか5分以内** という事実です。この短時間で技術力を効果的に示すには、適切な技術選択が不可欠となります。

## 1. 一般的な技術スタックの現状と選択理由

### 主流となっている技術構成

2024-2025年現在、エンジニアポートフォリオサイトで採用される技術スタックは大きく3つのカテゴリに分類されます。**静的サイトジェネレーター**（Jekyll、Hugo、Gatsby）は高速性と信頼性を重視する開発者に選ばれ、**React/Next.js**は約80%のプロジェクトで採用される事実上の標準 となっています。特に**Next.js 14/15**のApp RouterとTurbopackの組み合わせは、SSR/SSG機能とTypeScriptとの相性の良さから人気を集めています。

ホスティングプラットフォームについては、**Vercel**がNext.jsとの統合の容易さから第一選択となり、続いて**Netlify**、**GitHub Pages**が選ばれています。クラウドネイティブな選択としてAWSやCloudflareも増加傾向にあります。

### 技術選択の背景にある思考

技術スタックの選択理由として最も多いのは「**関連技術の実証**」です。履歴書に記載した技術スタックとポートフォリオで使用している技術の一致を採用担当者は厳重にチェックしており、これが技術選択の大きな動機となっています。 一方で、「**保守性とパフォーマンス**」も重要な要素で、特に画像最適化やCore Web Vitalsへの対応は必須要件となっています。

## 2. アーキテクチャ設計の複雑さレベル

### 一般的な設計パターン

ポートフォリオサイトのアーキテクチャは、目的によって3つのレベルに分類されます。**シンプルな静的サイト**は、HTMLとCSSを中心とした基本構成で、主にジュニアエンジニアや実用性重視の開発者が選択します。**SPAアプリケーション**は、ReactやVueを使用した動的なユーザー体験を提供し、フロントエンドスキルのアピールに適しています。**SSR/SSGハイブリッド**は、Next.jsを活用してSEOとパフォーマンスを両立させる、現在最も推奨される構成です。

採用担当者の調査によると、「**3-4クリック必要な凝ったウェブサイトは悪い例**」とされ、シンプルにプロジェクトの画像と説明文だけで十分という意見が多数を占めています。機械工学系企業の採用担当者は「150のウェブサイトを覚えておけない。印刷可能なPDFの方が良い」と明言しており、過度な複雑性は逆効果となることが明らかになっています。

## 3. 企業採用担当者が期待する技術レベル

### 重視される技術的要素

採用担当者が最も重視するのは「**実際に動作するプロジェクト**」です。GitHubでコードが確認でき、デモが見られることが必須条件となっています。 コードの可読性、テストとドキュメントの存在も、ジュニア開発者以上のレベルを示す重要な指標とされています。

技術的な評価基準として、**5分以内のレビュー時間** で判断されることを前提に、以下の要素が重視されます。 レスポンシブデザインは必須で、34%のポートフォリオがキーボードナビゲーション不可、40%が色彩コントラスト不足という調査結果から、アクセシビリティの重要性が浮き彫りになっています。

### 期待される実装品質

「**適切な技術選択**」が最も評価される要素です。標準的なエンジニア職では自作ウェブサイトは必須ではありませんが、スキルがあるなら良い印象を与えます。 重要なのは「全てのプロジェクトを含めるのではなく、関連性があり、最近で、注目に値するもののみを選択」 することです。

## 4. 高度なアーキテクチャパターンの実装事例

### DDD/Clean Architecture実装の実例

GitHubで公開されているポートフォリオサイトの中には、**DDD（Domain-Driven Design）**や**Clean Architecture**を採用した事例が存在します。`ddd-by-examples/library`や`kgrzybek/modular-monolith-with-ddd`などのプロジェクトは、複雑なビジネスロジックを持つドメインでの実装能力を示すために作成 されています。

これらの実装では、ビジネスロジックの明確化、テスト可能性の向上、技術的負債の削減というメリットがある一方、初期実装コストが高く、小規模なポートフォリオサイトには過度という評価も存在します。

### ヘキサゴナルアーキテクチャの採用例

`thombergs/buckpal`や`Sairyss/domain-driven-hexagon`などのプロジェクトは、ヘキサゴナルアーキテクチャを実装 しています。これらはビジネスロジックとインフラストラクチャの分離、高いテスト可能性、フレームワーク変更時の影響最小化を実現していますが、初期の複雑性増加と学習曲線の急峻さがデメリットとして挙げられています。

## 5. T3 Stack・tRPC・モノレポ構成の実装例

### T3 Stackの実装事例

**T3 Stack**（TypeScript、Next.js、tRPC、Prisma） を使用したポートフォリオサイトとして、`michu2k/t3-portfolio`が注目されています。管理ダッシュボード付きポートフォリオサイトとして、フルスタックType-safeアプリケーションを実現し、AWS S3を使用した画像アップロード機能も実装 されています。

tRPCによる型安全なAPI通信、Zodによる入力検証、Prismaによる型安全なデータベース操作という利点がある一方、学習コストの高さと特定の技術スタックへの依存がデメリット として指摘されています。

### モノレポ構成の採用

`t3-oss/create-t3-turbo`は、T3 StackとTurborepoを組み合わせたモノレポ実装の代表例です。Web・モバイル・APIの統合管理により開発効率性の向上を実現していますが、初期設定の複雑さと依存関係の管理負荷が課題 となっています。

## 6. オーバーエンジニアリングの境界線

### 「やりすぎ」と判断される技術選択

採用担当者の視点から、以下の実装は**オーバーエンジニアリング**と判断されます。静的コンテンツの表示にReact/Vue/Angularを使用すること、3つのクリック可能な画像を大きくするだけの要件に複雑なフレームワークを導入すること、 個人ポートフォリオでマイクロサービスアーキテクチャを採用することなどが典型例です。

**「Browser errors, over-engineering, inappropriate use of technology」**が採用担当者の主要な除外理由となっており、技術的な興味本位での実装は避けるべきとされています。「将来のデータベース変更に備えて」という理由でMicrosoft SQL ServerからOracleやNoSQLへの移行を想定した過度な依存性注入なども、不要な複雑性として批判されています。

## 7. アンダーエンジニアリングの境界線

### 「手抜き」と判断される実装

**基本的な品質要件の欠如**は致命的です。動作しないリンク、レスポンシブデザインの不備、アクセシビリティの無視などは、技術力不足の明確な証拠とされます。調査によると、50%の応募者がポートフォリオなしで応募し、34%のポートフォリオでキーボードナビゲーションが不可能という結果が出ています。

最低限必要な技術要素として、**HTML/CSS/JavaScriptの純粋なコードスキル**、**レスポンシブデザイン**（CSS Grid、Flexbox活用）、**WCAG AA準拠のアクセシビリティ**、**3秒以内のロード時間**が挙げられます。これらは技術レベルに関わらず、全てのエンジニアに求められる基本要件です。

## 8. 技術的複雑さと採用への影響

### ポジティブな評価を得る技術選択

採用にプラスの影響を与える技術選択は、**実際の問題解決**を示すものです。「前職で使われていたようなツールを再構築したアプリ」などの実用性の高いプロジェクトや、「生産時間を320時間削減し、出力を15%向上させた」といった定量的な成果を示すことが重要です。

**適切な技術選択**の例として、Next.js 14とTypeScript、Tailwind CSSの組み合わせは2025年の推奨スタックとなっています。JAMstack（JavaScript + APIs + Markup）による高パフォーマンスな実装や、Progressive Enhancementによる段階的な機能追加も評価されます。

### ネガティブな評価につながる要因

技術的な複雑さがマイナス評価となるのは、**目的と手段の不一致**が明確な場合です。ウェブ開発スキルをアピールしたいだけの過度な装飾、3-4クリック必要な複雑なナビゲーション、無関係な趣味プロジェクトの掲載などは、採用担当者に悪印象を与えます。

エンジニアリングマネージャーは「**優れたコードはジュニアエンジニアでも理解できる。素晴らしいコードは1年生でも理解できる**」と述べており、複雑さよりも明確さと保守性が重視されることを示しています。

## 結論：バランスの取れた技術選択への指針

エンジニアポートフォリオサイトの成功は、**目的の明確化**、**段階的実装**、**長期保守性**、**ユーザー体験**の4つのバランスにかかっています。2024-2025年のトレンドを踏まえつつ、採用担当者の5分間のレビュー時間で効果的に技術力を示すには、適切な複雑さと実用性のバランスが不可欠です。

最も重要なのは、技術選択に明確な理由を持つことです。「なぜこの技術を選んだのか」を説明でき、その選択が実際の価値を生み出していることを示すことが、優れたポートフォリオサイトの条件となります。技術的な興味だけでなく、採用担当者やユーザーの視点を考慮した実装が、最終的に最も高い評価を得ることになるでしょう。
