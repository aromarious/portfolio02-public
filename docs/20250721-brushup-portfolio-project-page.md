# ポートフォリオ技術説明ページ改善提案

**日付**: 2025-07-21  
**対象**: `/project-portfolio` ページ  
**視点**: ベテラン採用担当者レビュー

## 全体評価

**第一印象**: 技術説明が非常に丁寧だが、**候補者の実際の能力を判断するための情報が不足**している。

技術的には優秀だが、採用担当者が判断に必要な「人となり」「実力の証拠」「協働能力」の情報が決定的に不足している。

## 重大な不足点

### 1. 具体的な開発期間・工数が不明

**現状の問題**:

- いつからいつまで開発したのか不明
- 何人月かかったのか不明
- どの機能にどれくらい時間を要したのか不明

**採用担当者の懸念**:

- 開発速度が判断できない
- 工数見積もり能力が不明
- プロジェクト管理能力が分からない

**対応方針**: プロジェクト概要セクションを追加して開発期間・工数を明記する

### 2. 実際のコードが見えない

**現状の問題**:

- GitHubリポジトリへのリンクなし
- 具体的な実装例・コードスニペットなし
- 「本当にこの人が書いたのか？」が判断できない

**採用担当者の懸念**:

- 実装力の証明がない
- コード品質が確認できない
- 自分で実装したという証拠がない

**対応方針**: 「お問い合わせいただければリポジトリの閲覧権限をお渡しします」の案内を追加

### 3. 実務経験との関連性が曖昧

**現状の問題**:

- 「実務を前提とした」と言うが、実際の業務経験がどれくらいあるのか不明
- 過去の「JavaScript開発での困難」が具体的にどの案件・期間のことなのか不明

**採用担当者の懸念**:

- 実務経験の有無・深さが不明
- 業務での問題解決経験が分からない

**検証結果**: トップページに詳細な実務経験が記載済み（1995年〜、インターネット黎明期からの大規模システム開発・保守、メンタリング実績等）のため、この課題は実質的に解決済み

### 4. 問題解決の深さが不十分

**現状の問題**:

- 技術選択の理由は書いてあるが、「なぜその課題に直面したのか」の背景不足
- 「業務で求められる」と言うが、どの業務・業界を想定しているのか不明

**採用担当者の懸念**:

- 課題発見能力が不明
- 業界理解・ビジネス理解の深さが分からない

**対応方針**: 技術選択の背景により具体的なビジネス課題・業界想定を追加

## 改善が必要な点

### 1. 情報の整頓

**現状の問題**:

- 技術スタックの説明が長すぎる（フロントエンド・バックエンド・インフラの区分けが冗長）
- 「開発効率化」「品質管理」「自動化」の分類に実質的な意味がない

**改善案**:

- 技術スタック説明を簡潔化
- より意味のある分類軸を検討

### 2. 定量的データの信憑性

**現状の問題**:

- 「クエリ実行速度70%向上」「開発速度30%向上」の測定方法・基準が不明
- 何と比較してこの数値なのかが分からない

**改善案**:

- 測定方法・基準の明記
- 比較対象の具体的説明

### 3. アピール方法

**現状の問題**:

- 技術的価値の「※」印の項目が専門用語すぎて、非技術者には理解困難
- 差別化要因が本当に差別化になっているか疑問

**改善案**:

- 非技術者にも理解しやすい説明
- より明確な差別化要因の提示

**検証結果**: トップページに非技術者向けの説明・経験詳細が記載済みのため、技術説明ページとの役割分担で対応可能

## 良い点

### 1. 判断プロセスの明示

- 技術選択の比較検討が具体的
- 実装時の困難と解決策が実体験に基づいている

### 2. 実務意識

- 「3ヶ月後でも理解・修正できるコード」など、実務を意識した観点

## 採用担当者として最も知りたいこと

### 1. この人は即戦力になるか？

**現状**: 判断不可能  
**理由**: 実際の開発経験・期間が分からない

### 2. 技術的な問題解決能力はあるか？

**現状**: 部分的に示されている  
**改善点**: より具体的な事例が必要

### 3. チームで働けるか？

**現状**: 全く言及されていない  
**検証結果**: トップページにメンタリング実績・チーム統括経験・大規模開発参画経験が記載済み

### 4. 学習能力・適応力はあるか？

**現状**: 部分的に示されている  
**改善点**: 継続的学習の姿勢・新技術習得プロセス

## 具体的改善提案

### 1. プロジェクト概要セクション追加

**追加項目**:

- 開発期間（開始日〜終了日）
- 工数（稼働時間・人月）
- GitHubリポジトリリンク
- デモサイトリンク

**配置**: ページ上部、技術スタック説明の前

### 2. 具体的な実装事例セクション

**追加項目**:

- 重要な機能のコードスニペット
- 実際のトラブルシューティング事例
- パフォーマンス改善の具体例

**配置**: 技術選択セクションの後

### 3. 学習・成長プロセスセクション

**追加項目**:

- 新技術習得にかかった時間
- 試行錯誤の過程
- 失敗からの学び

**配置**: 設計原則セクションの後

### 4. チーム開発・協働経験セクション

**追加項目**:

- チーム開発でのコミュニケーション方法
- コードレビューでの学び
- 他者への技術指導経験

**配置**: 新規追加

### 5. プロジェクトの背景・動機セクション

**追加項目**:

- なぜこのプロジェクトを始めたのか
- どのような課題を解決したかったのか
- 対象ユーザー・想定シーナリオ

**配置**: 導入セクションの詳細化

## 優先順位

### 高優先度

1. プロジェクト概要（期間・工数・リポジトリ）
2. 具体的な実装事例
3. 定量的データの根拠説明

### 中優先度

4. 学習・成長プロセス
5. プロジェクト背景・動機
6. 情報整理・簡潔化

### 低優先度

7. チーム開発経験（実際の経験がある場合のみ）
8. アピール方法の改善

## まとめ

現在のページは技術的な詳細説明としては優秀だが、採用担当者が求める「候補者の人物像・実力・適性」を判断するための情報が不足している。

**トップページとの連携による課題解決**:

- 実務経験・メンタリング実績・チーム経験 → トップページで詳細記載済み
- 非技術者向け説明 → トップページで補完済み

**残る改善課題**:

1. **プロジェクト概要の追加**（開発期間・工数）
2. **リポジトリ閲覧案内の追加**
3. **定量的データの根拠説明**
4. **技術選択背景の具体化**（ビジネス課題・業界想定）

技術説明ページとトップページの役割分担を活かし、上記4点の改善により、より説得力のあるポートフォリオを実現できる。
